01Overview

Secrets Scanner is a defensive appsec tool that hunts hardcoded credentials wherever they hide: across source files, config, and — critically — the full git history, where a secret introduced in an old commit still lives and remains recoverable even after a later commit "removed" it. It is positioned as the defensive mirror of the offensive T1552.001 "Credentials in Files" technique, mapping to the AccuKnox SECURING SECRETS capability: one side finds the gap, this side closes it.

Its defining principle is that secrets never leak out of the scanner. Every finding carries only a redacted preview (for example `AKIA…8N (20 chars)`) and a stable SHA-256 fingerprint — the raw value is never written to a finding, log, baseline, or report. Detection is entirely read-only and never transmits anything. Rules are data, not code: all 28 signatures live in a single YAML pack that anyone can extend or tune without touching Python, with per-rule Shannon-entropy gating to keep generic patterns from firing on values like `password123`.

It is built to live in a pipeline. A single `--fail-on <severity>` flag turns any scan into a CI build gate; a baseline snapshot suppresses a repo's existing findings so the gate breaks only on a genuinely new leak; inline pragmas and an on-by-default allowlist keep noise down; and a shipped pre-commit hook plus a GitHub Actions workflow make adoption a two-line change. Findings roll up to CWE, OWASP Top 10, PCI DSS v4.0, and NIST 800-53 controls, so a scan doubles as GRC audit evidence.

The standout differentiator is safe, opt-in live verification. On demand, the `verify` command makes a single read-only request per unique secret to that secret's own provider and reports whether the credential is actually live — turning a wall of maybe-stale findings into a ranked "rotate this one now" list. It is engineered for safety: opt-in with confirmation, provider-scoped, no redirects (so the auth header can't leak), rate-limited and timed out, failing safe to UNVERIFIED on any network error, and never logging or storing the raw value.

02Key Capabilities

28 signature rules as tunable YAML

A single `rules/secret_patterns.yaml` pack detects cloud, VCS/CI, AI/LLM, SaaS, registry, and generic credential formats — extend or tune it without writing any Python.

Redact-by-construction findings

Every finding stores only a masked preview and a SHA-256 fingerprint, so the raw secret never reaches a log, report, baseline, or terminal.

Full git-history scanning

The `history` command walks every commit on every ref and scans added lines, catching a secret at the commit that introduced it even if a later commit deleted it — with author, email, date, and message for rotation.

Safe opt-in live verification

The `verify` command makes one read-only call per unique secret to its own provider and labels it VALID, INVALID, UNVERIFIED, or SKIPPED, so teams rotate confirmed-live keys first.

Per-rule entropy gating

Noisy or generic rules only fire above a configurable Shannon-entropy floor, cutting false positives on low-entropy placeholders like `password123`.

Opt-in entropy detector

The `--entropy` mode flags high-entropy, mixed-alphabet strings that no signature matches, catching unknown or novel key formats.

Baseline triage with incremental merge

Snapshot today's findings, then surface only new secrets; `baseline --update` folds triaged findings back in (with an optional reason) so review is incremental and fingerprints stay stable across line moves.

Three-layer noise control

An on-by-default allowlist drops doc samples and template placeholders, inline `# pragma: allowlist secret` comments suppress single lines, and `--gitignore` skips git-ignored files using git's own check-ignore engine.

CI fail-gate

`--fail-on <severity>` exits non-zero on any finding at or above a threshold, and `--fail-on-valid` breaks the build when a credential is confirmed live.

Multi-format reporting

`--report` writes JSON, CSV, or a self-contained dark-themed HTML report — with an executive summary, compliance roll-up, and findings table — inferring format from the file extension and with no Jinja2 dependency.

Compliance roll-up for GRC

Each finding is mapped to CWE-798/321/522, OWASP Top 10 (2021), PCI DSS v4.0, and NIST 800-53 controls, turning a scan into audit evidence.

Turnkey pre-commit and Actions integration

A shipped `.pre-commit-hooks.yaml` (id: secrets-scan) blocks commits introducing HIGH+ secrets, and the repo's own CI runs a self-scan gate across Python 3.10–3.12.

03Architecture

A small stdlib-first Python package built around a data pipeline: walk_files streams candidate files, a list of pluggable detectors emit redacted Findings, results are deduped by fingerprint and filtered through an optional baseline, then rendered as table/JSON or written as a report — with an optional fail-on gate. The history command swaps the file walk for a git-log diff walk that reconstructs each commit's added lines, and the verify command re-recovers raw secrets in-memory to make read-only provider calls. Detection rules and allowlist live entirely in YAML, so breadth grows without code changes.

1
Walker (core/walker.py)
Streams scannable files from a tree, skipping binaries (null-byte sniff + extension list), files over 5 MB, and noise dirs like .git, node_modules, venv, dist, and .terraform.
2
Detectors (detectors/)
A BaseDetector interface with two implementations: SignatureDetector (compiled YAML regex rules with capture groups, entropy gates, and verifier hints) and the opt-in EntropyDetector for novel high-entropy strings.
3
Scan engine (core/engine.py)
SecretScanner runs the detector set over each file's full content (so multi-line PEM keys match), applies inline-pragma and gitignore filtering, dedups by fingerprint, and drops baseline-suppressed findings into a ScanResult.
4
Git-history engine (core/git_history.py)
GitHistoryScanner replaces the file walk with a git log --all -p diff walk, reconstructing added lines per commit with a line map for accurate line numbers, and attributes each finding to the introducing commit.
5
Live verification (core/verify.py)
Verifier/Runner classes recover the raw secret in-memory from a finding's line and regex, then make a single no-redirect, timed, rate-limited read-only request per unique secret to its own provider (GitHub, GitLab, OpenAI, Anthropic, Stripe, SendGrid).
6
Models & safety core (core/models.py)
Severity/Verification enums, the Finding and ScanResult dataclasses, redact() masking, and the SHA-256 fingerprint() that powers dedup and baseline suppression without storing raw secrets.
7
Triage & suppression (core/baseline, allowlist, pragma, gitignore)
Baseline load/write/suppress with incremental --update, value/path allowlist from the rule pack, inline pragma line suppression, and git check-ignore-based filtering.
8
Reporting & compliance (core/reporter.py, core/compliance.py)
Dependency-free JSON/CSV/HTML report writers with redacted previews, plus a mapper that ties each rule to CWE/OWASP/PCI-DSS/NIST controls and rolls them up per scan.

05Security Controls

Cloud credentials
AWS Access Key ID (critical) and Secret Access Key, Azure Storage account key, and GCP API key, service-account JSON marker, and OAuth client secret.
VCS / CI tokens
GitHub classic PAT, fine-grained PAT, and OAuth/app/refresh/server tokens, plus GitLab PAT — all critical severity.
AI / LLM keys
OpenAI API key (entropy-gated at 3.5) and Anthropic API key, both critical and both live-verifiable.
SaaS providers
Stripe secret key, Slack token + incoming webhook, SendGrid, Twilio, Mailgun, Telegram bot, Shopify, and DigitalOcean tokens.
Registry tokens
npm access tokens and PyPI upload tokens, both critical.
Keys, JWTs & DB URIs
PEM private-key blocks (RSA/EC/OPENSSH/DSA/PGP), JSON Web Tokens, private-key variable assignments, and database connection URIs with inline credentials.
Generic assignments
Generic api_key/secret/token/password assignments gated by a 2.5 Shannon-entropy floor to suppress low-entropy placeholders.
Entropy detection
Opt-in detector flags high-entropy (>=4.0), mixed-alphabet strings of 24+ chars that no signature matches, catching unknown formats.
Live credential validation
Read-only provider checks for GitHub, GitLab, OpenAI, Anthropic, Stripe, and SendGrid classify a secret as VALID/INVALID/UNVERIFIED/SKIPPED, failing safe on any network error.
Leak-in-history detection
History scanning surfaces secrets that were committed then deleted but remain recoverable in git, attributed to the introducing commit and author for rotation.

07Quick Start

$ pip install -r requirements.txt   # or: pip install -e ".[test]"
$ python main.py scan --path .                         # scan a directory or file
$ python main.py scan --path . --fail-on high          # CI gate: exit 1 on HIGH+
$ python main.py baseline --path . -o .secrets-baseline.json   # snapshot existing findings
$ python main.py history --path . --max-commits 500    # scan full git history
$ python main.py verify --path . --yes --fail-on-valid # CI: break the build on a live key
