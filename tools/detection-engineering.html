01Overview

Detection Engineering is a production-grade library of Splunk SIEM detection content for security operations centers. It is not a scanner or an app — it is 468 hand-authored SPL correlation rules, each a self-contained YAML artifact carrying its detection query, MITRE ATT&CK mapping, severity and confidence scoring, false-positive tuning guidance, analyst investigation queries, and a step-by-step incident-response playbook. The goal is to close the gap between 'we collect the logs' and 'we would actually catch the attack,' shipping deploy-ready detections instead of theory.

Coverage spans the estates a real SOC has to defend: Windows endpoints across the full ATT&CK kill chain, Active Directory attack paths (BloodHound, DCShadow, RBCD, Shadow Credentials, Skeleton Key), the high-value credential-access attacks (Kerberoasting, Golden Ticket, DCSync, LSASS dumping, NTDS.dit extraction, Pass-the-Hash, NTLM relay, AD CS ESC1-ESC13), RHEL Linux across eight tactics via auditd and Sysmon-for-Linux, Apache web attacks, ICS/OT protocols, four major firewall platforms, three EDR products, SAP NetWeaver, and npm/GitHub supply-chain worms.

It also tracks what is happening right now: bespoke rule packs for active campaigns and freshly disclosed CVEs, including APT28's Operation NeuSploit (CVE-2026-21509), the China-nexus UAT-8837 critical-infrastructure intrusion, the June 2026 Patch Tuesday set, and the self-replicating Shai-Hulud / TeamPCP supply-chain worm — each built from real threat-intelligence reporting with IOCs, C2 indicators, and cross-product signatures (ClamAV, Snort SIDs).

The audience is detection engineers, SOC analysts, and blue teams running Splunk Enterprise / Splunk Cloud with Enterprise Security. Every rule is written to drop into a scheduled search or ES correlation search, is tagged to precise ATT&CK sub-techniques rather than just tactics, and comes with the operational context (prerequisites, audit-policy setup, lookup tables, quick-deploy ordering) that makes the difference between an alert that fires and an alert that gets tuned into silence.

02Key Capabilities

468 deploy-ready SPL rules

Hand-authored Splunk correlation searches across 12 platform and attack categories, each with a ready-to-run SPL query rather than pseudo-logic.

Precise MITRE ATT&CK mapping

Every rule is tagged to specific technique and sub-technique IDs (243 enterprise techniques plus 27 ATT&CK-for-ICS techniques) for coverage tracking and reporting.

Investigation queries per rule file

Each detection ships with follow-up SPL drilldowns so analysts can pivot from a notable event straight into triage without writing search-time SPL.

Built-in incident-response playbooks

Rules carry recommended_actions and optional remediation commands, turning a fired alert into a guided response procedure for the SOC.

False-positive tuning guidance

Documented known-FP sources, allowlist lookup tables, and report-mode baselining instructions so detections survive contact with a real environment.

Splunk ES notable events + risk scoring

Rules define notable titles with field substitutions, confidence (0-100) and risk scores, integrating cleanly with the Enterprise Security risk framework.

Active-campaign and CVE rule packs

Dedicated multi-rule packs for live threats — APT28 CVE-2026-21509, UAT-8837, Shai-Hulud npm worm, SAP CVE-2025-31324, and the June 2026 Patch Tuesday set.

IOC and cross-product enrichment

C2 IP/domain lists, file-hash lookups, and cross-tool signatures (ClamAV names, Snort SIDs) seed high-fidelity IOC-match detections.

Full ATT&CK kill-chain coverage on Windows & Linux

Detections span execution, persistence, privilege escalation, defense evasion, discovery, lateral movement, C2, collection, exfiltration and impact on both platforms.

Kill-chain correlation rules

Multi-stage rules chain related events (e.g., VSS creation plus ntds.dit copy, or LSASS access then NTLM logon) for the lowest-FP, highest-confidence alerts.

Consistent, self-documenting YAML schema

A uniform rule structure (name, description, mitre, severity, confidence, search, throttle, notable) makes rules reviewable, greppable, and portable.

Deployment prerequisites baked in

Each pack documents the exact audit policies, Sysmon/auditd config, SACLs, and lookup CSVs required before a rule can generate detections.

03Architecture

Detection Engineering is a declarative content library rather than a running application: each detection is a self-contained YAML document that bundles metadata, the SPL detection query, Splunk ES actions, tuning guidance, investigation drilldowns, and a response playbook. Rules are organized by platform and attack category under splunk_rules/, and are operationalized by copying their SPL into Splunk as scheduled or ES correlation searches, with CSV lookup tables providing environment-specific enrichment and allowlisting.

1
Rule definition (YAML schema)
Top-level rule_N_* keys declare name, description, mitre_attack, severity, confidence and risk_score — the reviewable, greppable metadata layer that also drives ATT&CK coverage tracking.
2
SPL detection query
The core `search:` block: a ready-to-deploy Splunk correlation search over the required index/sourcetype with stats, thresholds and lookups — the actual detection logic.
3
Notable event & risk layer
The `notable` block defines Splunk ES notable titles with $field$ substitutions, risk scores and confidence, wiring rules into the Enterprise Security risk framework.
4
Tuning & enrichment layer
Throttle windows, false-positive guidance, and CSV lookup tables (domain_controllers, approved_admin_ips, known-hash and IOC feeds) suppress noise and add context.
5
Investigation query layer
Per-file investigation_queries and drilldown searches give analysts pre-built pivots for triage after a rule fires.
6
Response playbook layer
recommended_actions and optional remediation_cmd fields turn each detection into a documented IR procedure for the responding analyst.
7
Category organization
The splunk_rules/ tree groups rules by platform/attack surface (windows, active_directory, credential_access, rhel_linux, apache_webserver, ics_ot, network_firewalls, edr_detection, recent_attacks, sap, supply_chain, network_security).

07Quick Start

$ git clone https://github.com/Krishcalin/Detection-Engineering — the repo is pure content; no build step is required.
$ Open a rule file (e.g. splunk_rules/credential_access/dcsync_attack_detection.yml) and copy a rule's `search:` SPL into Splunk > Search & Reporting.
$ Replace environment placeholders (e.g. YOURDC01$, YOUR-CA-01) and populate the referenced lookup CSVs (dc_ip_list.csv, approved_admin_ips.csv, IOC feeds).
$ Configure the documented prerequisites for that pack — audit policies, Sysmon/auditd rules, and SACLs — so the required events are generated.
$ Run the search over a lookback window in report mode to baseline, then save as a scheduled search or Splunk ES correlation search.
$ Deploy highest-fidelity rules first per each pack's quick-deploy ordering (e.g. tool-signature and IOC-match rules before broad behavioral rules).
