01Overview

RHEL-RT is an open-source, Python-based active security scanner purpose-built for red-team assessments of Red Hat Enterprise Linux 8 and 9. Instead of a generic Linux audit, every one of its 190 modules is written for the RHEL stack — SELinux, firewalld, auditd, PAM, FIPS mode, crypto policies, systemd sandboxing, and Podman/Docker — and every finding is mapped to a technique in the MITRE ATT&CK Linux Matrix so results speak the language security teams already use.

Safety is a first-class design decision. The scanner is read-only by default: modules run passive `check()` logic that never touches the target, and destructive or state-changing behavior is gated behind an explicit `--simulate` flag with reversible `cleanup()`. It scans the local host directly or reaches remote fleets over SSH (paramiko), so an operator can sweep a single box or a subnet of hardened servers from one workstation.

The output is built for action, not just observation. Findings render as dark-themed HTML with per-technique detail pages, export to JSON/CSV with compliance columns, and ship as an ATT&CK Navigator layer for heatmap visualization. Most notably, the tool auto-generates Ansible remediation playbooks — filtered by severity or compliance tag — that use proper Ansible modules rather than raw shell, turning an assessment into a fixable change set.

For a blue-team lead, a RHEL platform owner, or a red-teamer under authorized engagement, RHEL-RT closes the loop from ATT&CK-aligned detection to STIG/CIS/NIST compliance evidence to ready-to-run remediation — backed by a 570-test suite and a multi-version CI pipeline that keeps the check logic honest.

02Key Capabilities

Full ATT&CK Linux coverage

190 modules span all 12 ATT&CK tactics from Initial Access through Impact, each mapped to its technique ID for immediate framework alignment.

Safe-by-default scanning

Every module runs read-only passive checks unless the operator opts into active testing with an explicit --simulate flag backed by reversible cleanup.

Local and SSH remote scanning

Assess the local host directly or sweep remote RHEL fleets over SSH using paramiko with key or password authentication.

RHEL-native control checks

Purpose-built inspection of SELinux, firewalld, auditd, PAM, FIPS mode, crypto policies, and systemd sandboxing instead of generic Linux heuristics.

Container security modules

Ten dedicated Podman/Docker modules (CS001–CS010) cover runtime config, image trust, privilege escalation, seccomp/AppArmor, secrets, and supply chain.

Auto-generated Ansible remediation

The playbook generator emits proper Ansible-module remediation — not shell — filterable by severity or compliance tag, per-technique or consolidated.

ATT&CK Navigator export

Findings export as a color-coded Navigator JSON layer (red=vulnerable, green=secure) with severity scoring for instant heatmap visualization.

Multi-format reporting

Dark-themed HTML with clickable per-technique detail pages plus machine-readable JSON and CSV carrying compliance columns for all five frameworks.

Compliance cross-mapping

Each technique links to CIS Controls v8, NIST SP 800-53, CIS RHEL 9, and DISA STIG RHEL 8/9 controls to produce audit-ready evidence.

Module auto-discovery

Dropping a new module into modules/{tactic}/ registers it automatically — the engine discovers and loads it with no manual wiring.

Tunable scan profiles

Quick, full, and stealth profiles trade coverage for noise, letting operators pick rapid triage, comprehensive audit, or passive-only recon.

Evidence-chain logging

Structured logging records timestamp, target, technique ID, and result for each check to support defensible reporting.

03Architecture

RHEL-RT is a layered Python pipeline: a Click CLI drives an orchestration engine that auto-discovers technique modules, each module runs its checks through a session abstraction that transparently executes commands locally or over SSH, and the resulting findings flow into MITRE/compliance mappers, a Jinja2 reporting engine, and an Ansible playbook generator. All modules inherit a common BaseModule contract (check / simulate / cleanup / get_mitigations), which keeps the scanner, mapping, and reporting layers cleanly decoupled.

1
CLI & orchestration engine (main.py, core/engine.py)
Click-based command interface that auto-discovers modules, applies scan profiles, and orchestrates check/simulate runs across targets.
2
Session layer (core/session.py)
Unified execution abstraction that runs commands on the local host or a remote RHEL target over SSH via paramiko.
3
Technique modules (modules/, modules/base.py)
190 tactic-organized modules inheriting BaseModule, each implementing passive check(), gated simulate(), cleanup(), and get_mitigations().
4
Data models (core/models.py)
Typed Target, Finding, ModuleResult, Severity, Status, and Tactic structures that carry results through the pipeline.
5
Mapping & compliance (core/mitre_mapper.py, core/compliance.py)
Converts findings into an ATT&CK Navigator layer and cross-maps each technique to CIS, NIST, and DISA STIG controls.
6
Reporting engine (core/reporter.py, templates/report.html)
Renders dark-themed HTML with per-technique detail pages plus JSON and CSV exports using a Jinja2 template.
7
Remediation generator (core/playbook_generator.py)
Translates findings into Ansible playbooks using proper modules, with severity/tag filtering and per-technique or consolidated output.

05Security Controls

MITRE ATT&CK technique coverage
180 technique modules across 12 tactics: Initial Access (10), Execution (10), Persistence (17), Privilege Escalation (12), Defense Evasion (26), Credential Access (16), Discovery (26), Lateral Movement (8), Collection (14), Command & Control (18), Exfiltration (8), and Impact (15).
Discrete security checks
488 dedicated _check_* routines emit 1,043 distinct findings, each carrying a title, severity, evidence excerpt, and remediation string.
RHEL hardening controls
Targeted inspection of SELinux enforcement, firewalld, auditd rules, PAM configuration, FIPS mode, system crypto policies, and systemd service sandboxing.
Privilege escalation surface
SUID/SGID binaries, sudo misconfiguration, elevation-control (T1548), kernel-exploit exposure, systemd-service and cron privesc, and container escape-to-host (T1611).
Credential access
Checks on /etc/shadow, SSH keys, PAM auth modification, Kerberos tickets, unsecured credentials, credential stores, MFA interception, and network sniffing.
Persistence detection
Systemd services, boot/init scripts, cron and udev rules, XDG autostart, PAM backdoors, account manipulation, and binary compromise footholds.
Defense evasion
Rootkits, log/history tampering, PATH hijacking, obfuscated files, process injection, impair-defenses, hidden users and artifacts, and pre-OS-boot tampering.
Container security (CS001–CS010)
Podman/Docker runtime config, image trust, privilege escalation, network security, seccomp/AppArmor, resource limits, secrets management, filesystem, logging, and supply chain.
C2 and exfiltration exposure
Egress/DNS-tunneling and encrypted-channel indicators, protocol tunneling, plus exfil over C2, alternative protocols, physical media, and web services.
Compliance-mapped findings
Findings cross-map to CIS Controls v8, NIST SP 800-53 Rev. 5, CIS RHEL 9 Benchmark, and DISA STIG RHEL 8/9 rule IDs for audit-ready evidence.

07Quick Start

$ Passive local scan: python main.py scan --target localhost --profile quick
$ Remote SSH scan: python main.py scan --target 192.168.1.20 -u admin -k ~/.ssh/id_rsa --profile full
$ Active simulation (explicit opt-in): python main.py scan --target 192.168.1.20 --profile full --simulate
$ Scope by tactic or technique: python main.py scan --target localhost --tactic discovery  /  --technique T1082
$ Generate an HTML report: python main.py report --input reports/scan_abc123.json --format html
$ Generate Ansible remediation: python main.py remediate --input scan.json --severity critical --tags stig
