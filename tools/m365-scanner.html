01Overview

Microsoft 365 is the identity and collaboration backbone of most enterprises, yet its security posture is spread across a dozen admin portals — Entra ID, Conditional Access, Exchange, SharePoint, Teams, Intune, Defender, and Purview — each with its own settings that quietly drift out of policy. SSPM-O365 is an open-source, read-only SaaS Security Posture Management scanner that connects directly to a tenant through the Microsoft Graph API and evaluates 88 security rules across 24 check groups in a single pass, turning that scattered configuration surface into one prioritized, compliance-mapped findings report.

Unlike a checklist or a screenshot audit, the scanner performs a live assessment: it authenticates with OAuth 2.0 client-credentials (app-only, no user interaction), pages through Graph v1.0 and beta endpoints, and inspects the real state of MFA enforcement, Conditional Access, privileged roles, OAuth app consent, external sharing, guest access, audit logging, Identity Protection risk, Secure Score, and more. Every rule carries a severity, a CWE weakness class, the exact Graph endpoint it checked, a plain-English issue statement, and a concrete remediation step.

It is deliberately lightweight and safe to run against production. The entire tool is a single ~3,770-line Python file with one dependency (requests), makes only GET requests, and degrades gracefully when a permission or endpoint is unavailable. Findings are emitted to the terminal, JSON, or a self-contained filterable HTML report, and the process exits non-zero on CRITICAL/HIGH findings so it drops straight into a CI/CD gate.

The audience is M365 administrators, security engineers, and GRC teams who need continuous, evidence-backed assurance rather than a point-in-time consultant review. Because every one of the 88 rules is pre-mapped to CIS Microsoft 365 Foundations v3.1.0, NIST SP 800-53 Rev 5, ISO/IEC 27001:2022, and SOC 2 Trust Services Criteria, a single scan doubles as a compliance evidence artifact across four frameworks at once.

02Key Capabilities

Live Graph API scanning

Authenticates via OAuth 2.0 client credentials and queries Microsoft Graph v1.0 and beta endpoints to assess the tenant's real, current configuration rather than a static export.

88 rules across 24 check groups

Covers the full M365 stack — Entra ID, Conditional Access, MFA, privileged access, apps, guests, Exchange, SharePoint, OneDrive, Teams, audit, Intune, DLP, Defender, and governance — in one run.

Four-framework compliance mapping

Every finding is auto-enriched from a COMPLIANCE_MAP with its CIS M365 v3.1.0, NIST 800-53, ISO 27001:2022, and SOC 2 control IDs, so a scan is also an audit-evidence artifact.

Read-only and production-safe

Requires only Graph read permissions and issues exclusively GET requests, so it can never modify tenant state or lock users out.

Severity-ranked, CWE-tagged findings

Each result carries CRITICAL/HIGH/MEDIUM/LOW severity plus a CWE weakness class, a specific issue statement, and an actionable fix.

Multi-format reporting

Emits colorized terminal output, machine-readable JSON, and a self-contained dark-themed HTML report with severity counters, category/severity filters, and free-text search.

CI/CD-friendly exit codes

Returns exit code 1 when any CRITICAL or HIGH finding exists, making it a drop-in pipeline gate with a documented GitHub Actions example.

Graceful permission degradation

When a Graph endpoint returns 403/404 the scanner logs a verbose skip and continues, so a partial permission set still yields useful results instead of a crash.

Zero-install footprint

A single Python file with requests as its only dependency — no agents, no cloud services, no packaging — for trivial deployment and review.

'Verify:' findings for PowerShell-only settings

Where Graph cannot directly read a setting, the tool emits a Verify finding embedding the exact Exchange/PowerShell command to confirm the control manually.

Offline demo mode

run_demo.py injects synthetic findings across all categories to render sample terminal/JSON/HTML reports with no tenant or credentials required.

03Architecture

SSPM-O365 is a single self-contained Python script organized as a linear scan pipeline: authenticate, run 24 check-group methods that each call paginated Graph helpers, collect Finding objects that auto-enrich themselves with compliance metadata, then render to three output formats. There are no modules or packages by design — the single-file layout is an intentional deployment feature.

1
Reference data & compliance map
Module-level dicts — PRIVILEGED_ROLE_IDS (Entra role GUIDs), HIGH_RISK_PERMISSION_IDS (dangerous OAuth GUIDs), and COMPLIANCE_MAP (88 rule IDs to CIS/NIST/ISO/SOC2 controls) — that drive detection and enrichment.
2
Authentication layer
_authenticate() runs the OAuth 2.0 client-credentials flow against login.microsoftonline.com, caches the bearer token with expiry, and resolves the tenant display name.
3
Graph API access layer
_graph_get() and _graph_get_single() wrap Microsoft Graph v1.0 and beta endpoints with pagination, headers, and 403/404 graceful-skip error handling.
4
Check-group engine
24 self-contained _check_*() methods (security defaults, CA, MFA, privileged access, apps, guests, Exchange, SharePoint, OneDrive, Teams, audit, identity protection, secure score, Intune, DLP, Defender, sessions, cross-tenant, consent, auth strengths, governance, named locations, stale users) invoked in order by scan().
5
Finding model
The Finding data class captures rule_id, severity, the Graph endpoint checked, setting/value context, CWE, description, and recommendation, and auto-populates its compliance dict from COMPLIANCE_MAP on init.
6
Reporting layer
print_report(), save_json(), and save_html() render the collected findings to terminal, JSON, and a filterable self-contained HTML report, all including per-finding compliance data.
7
CLI entry point
main() parses arguments (tenant/client/secret, severity filter, json/html output, verbose) with argparse, supports M365_* env-var fallback, and sets the CRITICAL/HIGH exit code.

05Security Controls

Conditional Access & MFA
M365-CA (9 rules) and M365-MFA (4 rules) verify an enforced all-users MFA policy, legacy-auth blocking, risky sign-in and device-compliance policies, location policies, admin MFA, SMS-only MFA, and SSPR/MFA mismatch.
Privileged access
M365-PRIV (5 rules) flags excessive Global Admins, guest and service-principal admins, missing PIM, and break-glass account gaps using the PRIVILEGED_ROLE_IDS role map.
Application & OAuth governance
M365-APP (5) and M365-CONSENT (3) detect high-risk Graph permissions via HIGH_RISK_PERMISSION_IDS, expired/expiring secrets, secret-vs-certificate use, app sprawl, unrestricted user consent, and unverified-publisher apps.
Authentication strength
M365-AUTH (3 rules) audits phishing-resistant MFA (FIDO2/CBA) enforcement, authentication-strength binding in CA policies, and detection of weak SMS/Voice methods.
Identity Protection
M365-IDP (5 rules) surfaces high/medium-risk users, missing sign-in and user-risk policies, and risky service principals via the Identity Protection API.
Collaboration surface
M365-EXO (4), M365-SPO (4), M365-OD (3), and M365-TEAMS (3) check external mail forwarding, SMTP AUTH and legacy protocols, anonymous/external sharing links, default link type, OneDrive sync restrictions, and Teams anonymous meetings and external federation.
Endpoint & data protection
M365-INTUNE (4), M365-DLP (3), and M365-MDO (4) evaluate compliance-policy platform coverage and stale/non-compliant devices, sensitivity-label and DLP coverage, and Defender for Office 365 anti-phishing, Safe Links, Safe Attachments, and ZAP.
Logging, sessions & tenant isolation
M365-AUDIT (3), M365-SESSION (3), and M365-XTA (3) confirm Unified Audit Log and mailbox auditing, sign-in frequency and persistent-browser token policies, and cross-tenant MFA/device-trust and outbound B2B settings.
Governance & hygiene
M365-GOV (3), M365-LOC (3), and M365-STALE (3) check Access Reviews and Entitlement Management, missing or overly broad named locations, and inactive 90/180-day, never-signed-in, and disabled-but-privileged accounts.
Posture baselines
M365-SEC (2), M365-PWD (3), M365-GUEST (4), and M365-SCORE (2) assess Security Defaults/CA coexistence, password-expiry policy, guest invitation and staleness controls, and Microsoft Secure Score with unresolved high-impact improvement actions.

07Quick Start

$ pip install -r requirements.txt
$ Register an Entra ID app, add a client secret, grant the 19 read-only Graph application permissions, and grant admin consent
$ python o365_scanner.py --tenant-id YOUR_TENANT_ID --client-id YOUR_CLIENT_ID --client-secret YOUR_SECRET
$ export M365_TENANT_ID=... M365_CLIENT_ID=... M365_CLIENT_SECRET=... && python o365_scanner.py
$ python o365_scanner.py -t TENANT -c CLIENT -s SECRET --json report.json --html report.html --severity HIGH
$ python run_demo.py  # generate sample reports offline with no tenant

08Compliance & Frameworks

CIS Microsoft 365 Foundations Benchmark v3.1.0
Every rule maps to a CIS M365 section number, shown in terminal, JSON, and a dedicated HTML column.
NIST SP 800-53 Rev 5
Each finding carries the relevant NIST control IDs (e.g., IA-2(1)) in its compliance object.
ISO/IEC 27001:2022
Findings reference the applicable Annex A control (e.g., A.8.5).
SOC 2 Trust Services Criteria
Each rule is tagged with its SOC 2 TSC codes (e.g., CC6.1, CC6.2).
