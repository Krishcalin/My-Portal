01Overview

The MultiCloud Security Audit Tool is a ScoutSuite-inspired, Python-native auditing engine that connects to live AWS, Azure, and GCP accounts through their provider SDKs, evaluates real configuration state against a library of JSON rule files, and renders a single self-contained HTML report with posture scoring, compliance mapping, and remediation playbooks. It replaces per-cloud console clicking and spreadsheet triage with one repeatable command that speaks all three clouds.

The heart of the tool is a declarative rule engine borrowed from the ScoutSuite design philosophy: every security check is a small JSON file that names a data path (with `*` wildcards for iterating resources) and a conditions expression built from 32 leaf operators plus and/or/not logic. Because checks are data, not code, security teams can extend coverage, tune severities, or ship a custom ruleset without touching Python. The collector layer is strictly read-only, so scans never mutate cloud resources.

Findings are turned into decisions rather than noise. Each scan produces a 0-100 posture score with an A-F letter grade, an automatic compliance scorecard across CIS, PCI-DSS, HIPAA, SOC 2, NIST CSF, DORA, and AWS Well-Architected, and per-finding CLI fix commands that can be exported as a runnable remediation shell script. A YAML exceptions file suppresses false positives auditably (suppressed findings are greyed-out, not hidden).

Built for cloud security engineers, platform teams, and auditors, it is CI/CD-native out of the box: SARIF 2.1.0 feeds GitHub Code Scanning, JUnit XML turns CRITICAL/HIGH findings into pipeline gate failures, and a bundled GitHub Actions workflow scans all three clouds in parallel on every push. A zero-credential demo mode makes the whole pipeline explorable in one command.

02Key Capabilities

Unified tri-cloud auditing

One CLI (scout.py) audits live AWS, Azure, and GCP accounts against 98 rules spanning 37 services, using each provider's native SDK/API.

Declarative JSON rule engine

Every check is a JSON file with a data path and conditions expression, so coverage can be extended or tuned without writing Python.

30+ condition operators with path wildcards

The condition engine offers 32 leaf operators (null/empty, boolean, equality, string, regex, list, dict, length, network CIDR, date-age) plus and/or/not logic, evaluated over dot-notation paths with `*` resource iteration.

0-100 posture score with A-F grade

A weighted severity model (CRITICAL 40, HIGH 15, MEDIUM 5, LOW 1) produces an at-a-glance posture score and letter grade per scan.

Automatic compliance scorecard

Findings are mapped and aggregated to CIS Benchmarks, PCI-DSS, HIPAA, SOC 2, NIST CSF, DORA, and AWS Well-Architected.

Remediation playbooks

Per-finding CLI fix commands (56 templates across the three clouds) are embedded in the report and exportable as a runnable bash remediation script.

Auditable exception management

A YAML suppressions file silences false positives by rule/resource with a reason and expiry, and suppressed findings appear greyed-out in the report rather than being hidden.

SARIF 2.1.0 export

Findings export to SARIF for GitHub Code Scanning, GitLab, or VS Code, with CRITICAL/HIGH mapped to error severity.

JUnit XML CI gate

Findings become JUnit test results grouped by provider.service, where CRITICAL/HIGH become failures to break a pipeline.

Self-contained HTML report

A single dark-themed HTML file with an SVG posture gauge, compliance cards, service matrix, and collapsible fix-command panels — no server or assets required.

Zero-credential demo mode

`scout.py demo` renders a full sample report with built-in findings so the entire pipeline can be evaluated without any cloud access.

CI/CD-native GitHub Actions workflow

The bundled scout-scan.yml runs parallel AWS/Azure/GCP scans on push, PR, and schedule, uploading SARIF, JUnit, and HTML artifacts and posting a PR summary.

03Architecture

The tool is a three-stage pipeline: provider collectors make read-only API calls and build a nested data dict, a ProcessingEngine walks each rule's path (expanding `*` wildcards) and evaluates its conditions expression to emit Findings, and an output layer post-processes findings into scores, compliance rollups, and multiple report formats. Rules are declarative JSON decoupled from collector code, and all three clouds share the same core engine and output modules.

1
Provider collectors (providers/)
Per-cloud facades (AWSFacade via boto3, AzureFacade via Azure SDK + Microsoft Graph, GCPFacade via the Discovery API) and service fetchers make read-only calls and return a nested data dict shaped as {service: {resource_type: {name: flags}}}.
2
Ruleset loader (core/rule.py, core/ruleset.py)
Ruleset parses the ruleset JSON, instantiates enabled Rule objects from individual JSON rule files, and resolves _ARG_N_ token substitution and _INCLUDE_ references.
3
Condition engine (core/conditions.py)
pass_conditions() evaluates a rule's conditions against each item using 32 leaf operators plus and/or/not logic; returns True when an item is flagged.
4
Processing engine (core/engine.py)
ProcessingEngine.run() walks each rule.path through the data dict (with `*` wildcard expansion and nested indexing), applies conditions per item, and produces a sorted list of Finding objects.
5
Analytics (core/scoring.py, core/compliance.py, core/exceptions.py)
Computes the 0-100 posture score and A-F grade, aggregates per-framework compliance counts, and applies auditable YAML-based suppressions.
6
Output layer (output/)
Serializes findings into a self-contained HTML report, SARIF 2.1.0, JUnit XML, JSON, and a bash remediation script from CLI fix-command templates.
7
CLI orchestrator (scout.py)
Argparse subcommands (demo/aws/azure/gcp) wire provider selection, engine run, exception application, scoring, and all output writers, returning exit codes for CI gating.

05Security Controls

AWS coverage (37 rules / 16 services)
IAM (root MFA/access keys, user MFA, weak/no-expiry password policy, stale keys), S3 (public access block, encryption, versioning, logging), EC2 (open SSH/RDP SGs, IMDSv2, unencrypted EBS), RDS (public, unencrypted, no backup/deletion protection), CloudTrail (not logging, not multi-region, no log validation), KMS rotation, GuardDuty, Config, VPC flow logs, Lambda, EKS, ECR, Secrets Manager, and OpenSearch.
Azure coverage (33 rules / 12 services)
Entra ID (MFA gaps, legacy auth, guest users, security defaults), Storage (HTTPS-only, min TLS, public blob, soft delete), Key Vault (purge protection, soft delete, public network, diagnostics), Network (open SSH/RDP), SQL (auditing, threat detection, Entra admin), Defender for Cloud, Monitor log retention, App Service, AKS (RBAC/AAD/network policy), Container Registry, and Cosmos DB.
GCP coverage (28 rules / 9 services)
IAM (primitive roles, service-account admin roles, stale SA keys), Cloud Storage (public access, uniform access, versioning, logging), Compute (public IP, OS Login, serial port, default network/SA, full-API SA scope, open SSH/RDP firewall), Cloud SQL (public IP, SSL required, backups), Logging sinks/audit logs, KMS rotation, GKE, BigQuery, and Cloud Functions.
Severity model
The 98 rules are graded 16 CRITICAL, 35 HIGH, 41 MEDIUM, and 6 LOW, driving both the posture-score penalties and the CI gate exit codes.
Public-exposure detection
Network operators (ispubliccidr, insubnets/notinsubnets) evaluate CIDR reachability to flag publicly-open security groups, firewall rules, public IPs, and public storage/registry/database endpoints.
Encryption & key hygiene
Checks enforce encryption at rest (EBS, RDS, S3, OpenSearch, BigQuery CMEK, EKS secrets) and automatic KMS key rotation across AWS, Azure, and GCP.
Credential & identity hardening
MFA enforcement, root/privileged account controls, weak password policy, and stale-credential detection via date-age operators (olderthandays) for access keys and service-account keys.
Logging & audit assurance
Verifies CloudTrail/Config, Azure Monitor/diagnostics, and GCP audit logging + sinks are enabled and retained so detective controls are actually in place.
Auditable exception suppression
YAML suppressions match by rule/resource/service with a reason and expiry; expired or non-matching suppressions never silence a finding, and suppressed items remain visible in reports and SARIF.
Read-only guarantee
All provider fetchers make only read/list/describe API calls and never modify cloud resources, making scans safe to run against production accounts.

07Quick Start

$ git clone https://github.com/Krishcalin/MultiCloud-Security-Audit-Tool.git && cd MultiCloud-Security-Audit-Tool && pip install -r requirements.txt
$ python scout.py demo --html report.html   # zero-credential sample report
$ python scout.py aws --region us-east-1 --html aws_report.html
$ python scout.py azure --subscription-id <SUB_ID> --html azure_report.html
$ python scout.py gcp --project <PROJECT_ID> --sarif results.sarif --junit results.xml
$ python scout.py aws --exceptions exceptions.yaml --remediation-script fix.sh
