01Overview

CrowdStrike Falcon is one of the most capable EDR platforms on the market, yet its protection is only as strong as its configuration. Detect-only prevention policies, broad ML exclusions on temp folders, LOLBin process exclusions, sensors stuck in Reduced Functionality Mode, and hosts with no prevention policy assigned are all silent gaps that attackers exploit long after the license is paid for. The CrowdStrike Falcon Validation Scanner audits your real Falcon configuration exports against CrowdStrike's recommended settings, industry best practices, and known attacker techniques to surface those gaps before an adversary does.

The scanner runs entirely offline. It never touches the Falcon API or a live tenant; instead it ingests JSON exports (from the FalconPy SDK, the Falcon console, or Falcon Data Replicator) and evaluates them through 10 audit modules spanning prevention policy, sensor update policy, response/RTR, device control, exclusions, sensor health, admin & API security, custom IOAs, firewall, and MITRE ATT&CK coverage. Findings are severity-ranked (Critical/High/Medium/Low), mapped to stable check IDs, and rendered into an interactive HTML dashboard with a weighted risk score.

It is built for the people who own Falcon day-to-day: EDR administrators, detection engineers, SOC leads, and security auditors who need a repeatable, evidence-based way to prove their deployment is actually configured to block. Because it is pure-Python with zero third-party dependencies and ships with a sample dataset full of deliberate misconfigurations, it runs anywhere Python 3.8+ is available and can be dropped into a monthly configuration-drift review with a single command.

What makes it notable is its focus on the Exclusion Audit module - the single most dangerous and least-reviewed area of an EDR deployment. It flags ML, IOA, and Sensor Visibility exclusions that land on attacker-abused paths, wildcard patterns, and living-off-the-land binaries, calling out the difference between an ML exclusion (blind to one engine) and a Sensor Visibility exclusion (a complete telemetry blind spot). This is the layer where real breaches hide.

02Key Capabilities

Prevention Policy Validation (10 checks)

Inspects every prevention policy for disabled NGAV/ML engines, behavioral IOA gaps, missing exploit mitigations, detect-only mode, ransomware protection, script monitoring, and unprotected or unassigned hosts.

Exclusion Audit - the most critical module

Flags ML, IOA, and Sensor Visibility exclusions on attacker-abused paths, wildcard patterns, LOLBin processes, and global-scope or excessive exclusions that quietly negate detection.

Sensor Health & Coverage

Detects offline/degraded sensors, hosts in Reduced Functionality Mode (near-zero detection), stale sensors offline 30+ days, and reports OS fleet distribution for coverage analysis.

Admin & API Security

Surfaces excessive Falcon Administrator accounts, console users without MFA, OAuth2 API clients with write/admin/wildcard scopes, and default-only RBAC roles.

Sensor Update Policy audit

Catches pinned sensor versions, fragmented version sprawl across the fleet, and disabled uninstall/maintenance protection that lets local admins remove the sensor.

MITRE ATT&CK coverage assessment

Maps the top 10 attacker techniques (T1059, T1055, T1486, T1218 and more) against prevention-policy settings to expose behavioral detection gaps.

Response & Device Control checks

Verifies Real-Time Response availability, flags unrestricted custom RTR scripts, and detects missing or default-ALLOW USB device control policies.

Interactive HTML dashboard report

Generates a self-contained, dark-themed HTML report with a weighted risk score, severity and category breakdowns, and per-finding remediation guidance and references.

Fully offline, air-gap friendly

Analyzes only local JSON exports and never connects to the Falcon API or a live environment, making it safe for regulated and air-gapped assessments.

Flexible export ingestion

A resilient data loader recognizes 19 config file types with filename aliases from FalconPy, the Falcon console, or Falcon Data Replicator, tolerating both list and resources-wrapped JSON shapes.

Targeted and severity-filtered scans

Run all 10 modules or a subset (e.g. only exclusions), and filter output by minimum severity to focus remediation on Critical findings first.

Batteries-included sample dataset

Ships with 13 demo config exports containing deliberate misconfigurations plus a pre-generated report so users can see real findings immediately.

03Architecture

A single CLI entry point (cs_scanner.py) loads all Falcon JSON exports through a DataLoader, then dispatches them to 10 auditor classes registered in a MODULE_MAP. Each auditor extends a shared BaseAuditor that standardizes finding records (check ID, severity, category, affected items, remediation, references). Aggregated findings are optionally severity-filtered and passed to an HTML ReportGenerator, with graceful JSON fallback if the reporter is unavailable.

1
CLI orchestrator (cs_scanner.py)
Parses arguments (--data-dir, --modules, --severity, --output), prints the banner, dispatches selected modules via MODULE_MAP, aggregates and severity-filters findings, and prints the Critical/High/Medium/Low summary.
2
Data loader (modules/base.py)
DataLoader resolves 19 config file types via a FILE_MAP with filename aliases, reads UTF-8/BOM JSON, and reports how many config files loaded successfully.
3
BaseAuditor contract
Shared base class defining severity constants and a finding() factory that produces normalized finding dicts (check_id, title, severity, category, affected_items/count, remediation, references, timestamp).
4
Policy validation engines (modules/policy_validation.py)
Modules 1-4: PreventionPolicyAuditor, SensorUpdateAuditor, ResponsePolicyAuditor, and DeviceControlAuditor validate policy configuration against recommended defaults.
5
Advanced validation engines (modules/advanced_validation.py)
Modules 5-10: ExclusionAuditor, SensorHealthAuditor, AdminSecurityAuditor, CustomIoaAuditor, FirewallPolicyAuditor, and MitreCoverageAuditor cover exclusions, fleet health, identity, and ATT&CK mapping.
6
Report generator (modules/report_generator.py)
ReportGenerator computes a weighted risk score and category/severity breakdowns, then renders a self-contained interactive dark-themed HTML dashboard.

05Security Controls

Prevention posture (NGAV/ML)
Detects disabled Cloud/Sensor Anti-Malware, Adware/PUP, and ML sliders set to Cautious/Disabled instead of Moderate/Aggressive that let zero-days bypass the ML engine (PREV-002, PREV-005).
Detect-only vs block enforcement
Flags policies where behavioral settings detect but do not block, and ransomware protection in detect-only/disabled mode - the #1 post-deployment gap (PREV-006, PREV-009).
Behavioral & exploit prevention
Checks Suspicious Processes, Registry Ops, Scripts, Intel Threats, Kernel Drivers, script-based execution monitoring, and memory mitigations ASLR/DEP/Heap-Spray/Null-Page/SEH (PREV-003, PREV-004, PREV-010).
Coverage & policy assignment
Identifies hosts with a sensor but no prevention policy (completely unprotected) and prevention policies not assigned to any host group (PREV-007, PREV-008).
Dangerous exclusions
Audits ML/IOA/SV exclusions against 17 attacker-abused paths, 14 executable extensions, wildcard patterns, global scope, and excessive counts - distinguishing ML blind spots from total Sensor Visibility telemetry blindness (EXC-001 through EXC-006).
LOLBin / living-off-the-land exclusions
Critical check for exclusions covering 14 commonly abused binaries (powershell.exe, cmd.exe, mshta.exe, certutil.exe, regsvr32.exe, bitsadmin.exe, bash, python and more) (EXC-007).
Sensor health & RFM
Flags sensors not in normal status, hosts in Reduced Functionality Mode (near-zero detection after kernel drift), and stale sensors offline 30+ days (SENSOR-001 to SENSOR-003).
Sensor update & tamper protection
Detects pinned/legacy sensor versions, fleet version sprawl, and disabled uninstall/maintenance protection that permits silent sensor removal (UPD-002 to UPD-004).
Admin, MFA & API scope hygiene
Surfaces excessive Falcon Admins, console users without MFA, OAuth2 API clients with write/admin/wildcard scopes, and default-only RBAC (ADMIN-001 to ADMIN-004).
MITRE ATT&CK coverage
Maps top 10 techniques (T1059, T1053, T1003, T1021, T1055, T1078, T1486, T1082, T1047, T1218) to required Falcon settings and reports coverage gaps (MITRE-001).

07Quick Start

$ git clone https://github.com/Krishcalin/CrowdStrike-Falcon-Validation-Scanner.git && cd CrowdStrike-Falcon-Validation-Scanner
$ python cs_scanner.py --data-dir ./sample_data --output report.html  # scan bundled sample data with deliberate misconfigurations
$ python cs_scanner.py --data-dir ./exports --modules prevention exclusions sensors  # scan specific modules
$ python cs_scanner.py --data-dir ./exports --severity HIGH  # show only High and above
$ python cs_scanner.py --data-dir ./exports --modules exclusions  # audit only the most critical module
