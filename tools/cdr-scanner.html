01Overview

Cloud Detection & Response (CDR) Scanner is a zero-dependency, single-file Python engine that analyses exported cloud audit logs -- AWS CloudTrail, Azure Activity/Sign-in, and GCP Cloud Audit -- to surface active threats, backdoors, and misconfigurations mapped to the MITRE ATT&CK Cloud Matrix. Traditional SIEM rulepacks systematically under-cover cloud-native attacks: root console logins, IAM backdoor users, CloudTrail/GuardDuty tampering, public S3/GCS buckets, and cross-account role abuse. CDR closes that gap with a curated catalogue of provider-specific detections that read the raw audit event and reason about it, not just grep for an event name.

Under the hood the scanner auto-detects the cloud provider from each log file's JSON shape, unwraps the many envelope formats each vendor emits (CloudTrail `Records`, Azure `value`/`records`, GCP `entries`), extracts a normalised field set, and evaluates every matching detection rule. Rules go beyond pattern matching by attaching Python lambda condition functions -- so 'ConsoleLogin' only fires a finding when the caller is Root, when MFA was absent, or when the login failed. Each finding carries its MITRE technique IDs, mapped compliance controls, actor, source IP, region, and a concrete remediation recommendation, and is rendered to a coloured console report, machine-parseable JSON, or an interactive dark-themed HTML dashboard.

The whole tool is one portable Python file requiring only the 3.10+ standard library -- no pip installs, no cloud SDKs, no network calls -- so it runs identically on an analyst laptop, an air-gapped forensics box, or a CI runner. A non-zero exit code on CRITICAL/HIGH findings makes it a natural deployment gate.

It is built for cloud security engineers, incident responders, and detection engineers who need a fast, offline, transparent way to triage a bundle of cloud logs after an alert, and for platform teams who want a lightweight guardrail wired into their pipeline.

02Key Capabilities

Unified multi-cloud detection

One engine detects threats across AWS CloudTrail, Azure Activity/Sign-in, and GCP Cloud Audit logs, each with its own dedicated rule set and field extractor.

60 curated detection rules

36 AWS, 14 Azure, and 10 GCP rules spanning nine ATT&CK tactics from Initial Access through destructive Impact.

Automatic provider detection

`_detect_cloud` fingerprints each file from its JSON structure (eventSource, operationName, protoPayload) so no provider flag or configuration is ever needed.

Contextual lambda conditions

18 helper functions (e.g. `_ct_mfa_used`, `_ct_cross_account`, `_ct_sg_open`, `_az_global_admin`, `_gcp_public_bucket`) reason about event context instead of matching event names alone, cutting false positives.

MITRE ATT&CK Cloud Matrix mapping

Every finding is tagged with ATT&CK technique IDs resolved against a 40-technique reference map, and reports include a per-tactic detection summary.

Compliance context on findings

Findings are annotated with the CIS AWS/Azure/GCP, NIST 800-53, SOC 2, and PCI-DSS controls they violate for audit-ready output.

Three report formats

ANSI-coloured console output, indented JSON for SIEM/SOAR ingestion, and a self-contained interactive HTML dashboard -- generated in a single run.

Interactive HTML dashboard

Dark-themed report with severity summary cards, a MITRE tactic breakdown, and live client-side severity checkboxes to filter the findings table.

CI/CD-friendly exit codes

Returns exit code 1 when any CRITICAL or HIGH finding is present and 0 otherwise, so it can gate a pull request or pipeline stage.

Zero-dependency single file

The entire scanner is one ~1,100-line Python file using only the 3.10+ standard library -- portable, auditable, and installable by copying one file.

Severity model and filtering

Five-level CRITICAL/HIGH/MEDIUM/LOW/INFO scale with a `--severity` floor to focus a scan on only the findings that matter.

Robust envelope parsing

Dedicated parsers accept single events, vendor batch envelopes, or raw arrays, and the scanner silently skips malformed files and events so one bad log never aborts a scan.

03Architecture

CDR is a single-file, staged pipeline. Each log file flows through provider auto-detection, an envelope parser, a field extractor, and the rule engine, which evaluates every matching detection's lambda condition to emit Finding dataclasses; those findings are then serialised by three interchangeable reporters. All rules, engine logic, and report templates live in cdr_scanner.py with no external dependencies.

1
Detection rule catalogue
Module-level `AWS_RULES` (36), `AZURE_RULES` (14), and `GCP_RULES` (10) dicts plus the `MITRE_MAP` (40 techniques) and `COMPLIANCE_MAP` (7 frameworks) lookup tables that drive every finding's metadata.
2
Condition helpers
Cloud-scoped lambda functions prefixed `_ct_` (AWS), `_az_` (Azure), and `_gcp_` (GCP) that perform contextual analysis -- MFA checks, cross-account role math, 0.0.0.0/0 exposure, public-ACL detection, GPU-instance cryptomining heuristics.
3
Parsers & auto-detection
`_parse_cloudtrail`, `_parse_azure_log`, `_parse_gcp_log` unwrap vendor JSON envelopes, and `_detect_cloud` fingerprints the provider from log structure before routing.
4
Field extractors
`_extract_aws_fields`, `_extract_azure_fields`, `_extract_gcp_fields` normalise each provider's schema into a common event_time / event_name / actor / source_ip / region shape consumed by the rule engine.
5
CDRScanner engine
Orchestrates `scan_path` -> `_scan_directory` -> `_scan_file` -> `_analyse_events` -> `_add_finding`, matching event names, running conditions, and accumulating findings with cloud and tactic tallies.
6
Reporting layer
`print_report` (coloured console with tactic summary), `save_json` (structured export), and `save_html` (interactive dark dashboard) render the same Finding list three ways.
7
CLI & exit gate
An `argparse` front end exposes target/--json/--html/--severity/--verbose/--version and converts CRITICAL/HIGH findings into a non-zero exit status.

05Security Controls

Root & admin account abuse (Initial Access)
Detects AWS root console logins, console logins without MFA, brute-force login failures, and API calls from unusual regions; Azure risky sign-ins, failed sign-in bursts, and MFA-absent logins.
Persistence / backdoors
Flags new IAM users, access keys, login profiles, Lambda functions and EC2 key pairs on AWS; new/guest users, app credentials and OAuth consent grants on Azure; and service-account and SA-key creation on GCP.
Privilege escalation
Catches AdministratorAccess and wildcard (Action:*/Resource:*) IAM policy attachments, inline policies, and cross-account role assumption on AWS; Global Administrator and privileged directory-role assignments on Azure; roles/owner and roles/editor and custom-role grants on GCP.
Defense evasion / log tampering
Detects CloudTrail stopped or deleted, GuardDuty disabled, Config recorder stopped, and VPC flow logs deleted on AWS; Conditional Access and Defender disabled and diagnostic settings deleted on Azure; audit sink and firewall-rule deletion on GCP.
Credential access
Surfaces Secrets Manager retrieval, SSM parameter access, and password-policy weakening on AWS, mapped to T1552 Unsecured Credentials and T1556 Modify Authentication Process.
Discovery / reconnaissance
Identifies enumeration and infrastructure-discovery API patterns and S3 bucket listing, mapped to T1580 Cloud Infrastructure Discovery and T1619 Cloud Storage Object Discovery.
Data exfiltration
Detects S3 buckets and GCS buckets made public (public-read / allUsers / allAuthenticatedUsers), EBS snapshots shared with all, and RDS snapshots shared externally (T1530, T1537).
Network exposure
Flags security groups and firewall rules opened to 0.0.0.0/0 or ::/0, deleted SG rules, VPC modification on AWS, NSG any-source rules on Azure, and open firewalls on GCP.
Destructive impact
Catches EC2 termination, RDS/database deletion, S3/bucket and resource-group deletion, KMS key deletion or destruction, and Key Vault secret deletion across all three clouds (T1485 Data Destruction, T1578.003).
Cryptomining heuristic
`_ct_crypto_instance` flags launches of GPU/accelerator and oversized instance families (p3/p4/p5/g4dn/g5/inf/trn and 24xlarge/32xlarge) as likely resource hijacking (T1496).

Python 3.10+ (uses `from __future__ import annotations`, PEP 604 typing)
Dependencies
None -- pure standard library (argparse, json, dataclasses, pathlib, re, os, datetime)

Declarative rule dicts + Python lambda condition functions for contextual analysis

Console (ANSI colour), JSON, and self-contained interactive HTML (inline CSS/JS, dark theme)
Testing

CI/CD

Deployment
Single portable file -- clone the repo or curl cdr_scanner.py; runs on Windows, macOS, Linux

07Quick Start

$ git clone https://github.com/Krishcalin/Cloud-Detection-Response.git && cd Cloud-Detection-Response
$ python cdr_scanner.py --version   # verify: CDR Scanner v1.0.0
$ python cdr_scanner.py /path/to/cloud-logs/   # scan a directory of AWS/Azure/GCP logs
$ python cdr_scanner.py ./logs --json report.json --html report.html   # emit both reports
$ python cdr_scanner.py ./logs --severity HIGH   # only CRITICAL and HIGH findings (CI gate)
$ python cdr_scanner.py tests/samples/ --verbose   # run the bundled malicious-log corpus
