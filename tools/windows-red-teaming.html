01Overview

Windows Red Teaming is an open-source, MITRE ATT&CK-aligned assessment framework for authorized red team engagements against Windows 10, 11, Server 2019, and Server 2022. Every check maps to an ATT&CK technique, and the tool pairs two complementary engines under one CLI: 50 hand-written Python audit modules that perform deep, read-only security-posture checks, and a 226-test library of Atomic Red Team-style YAML tests that execute real adversary behavior with input arguments, dependencies, and automatic cleanup. Together they span 12 ATT&CK tactics from Reconnaissance through Impact.

Its defining trait is safe-by-default, dual-mode execution. Analysts can run a fully passive `check` scan that is production-safe, preview exactly what would run with `--dry-run` (zero target contact), or opt in to active `--simulate` mode where Python modules and YAML atomics execute technique behavior and then automatically revert it. An OS-guard auto-skips modules that don't apply to the target, an authorization banner is confirmed before every scan, and every action is written to a timestamped evidence chain for full attribution.

The tool is aimed at red teamers, purple teamers, and detection engineers who need repeatable, attributable technique execution mapped to a framework rather than ad-hoc scripts. It exports ATT&CK Navigator layers to visualize coverage, generates HTML/JSON/CSV reports, and maps findings to CIS Benchmark v8 and NIST 800-53 Rev 5 controls so results land equally well with a SOC or a compliance auditor.

Notably, it ships modules and atomic tests for four February 2026 Patch Tuesday CVEs (RDP EoP, DWM SYSTEM privesc, MSHTML and SmartScreen security bypasses), and its auto-discovery design means dropping a new Python module into modules/<tactic>/ or a YAML file into atomics/<technique>/ extends coverage with no code changes to the engine.

02Key Capabilities

Dual-mode scanning engine

A single ScanEngine orchestrates 50 Python posture-audit modules and 226 YAML atomic tests, so one scan covers both deep audit logic and broad technique execution.

Check / simulate / dry-run modes

Passive read-only check mode is safe for production, --simulate actively executes techniques with rollback, and --dry-run previews the exact plan with zero target contact.

Atomic Red Team-style YAML tests

Data-driven tests support #{arg} templating with defaults and CLI overrides, prereq/get-prereq dependencies, and three executor types (powershell, command_prompt, manual).

Auto-discovery of modules and atomics

Dropping a Python module in modules/<tactic>/ or a YAML file in atomics/<technique>/ is picked up automatically with no changes to the engine.

Guaranteed cleanup and evidence chain

Both Python cleanup() and YAML cleanup_command run after every simulation (even on error), while every action is logged with timestamp, target, technique ID, and result.

ATT&CK Navigator layer export

MitreMapper emits ATT&CK Navigator v4.5 JSON layer files so coverage and results can be visualized directly on the MITRE technique matrix.

Multi-format reporting

Reporter renders a dark-themed Jinja2 HTML report with executive summary plus machine-readable JSON and CSV outputs for pipelines and SIEMs.

Compliance mapping

compliance_mapper translates ATT&CK findings into CIS Benchmark v8 and NIST 800-53 Rev 5 controls for audit-ready reporting alongside red team results.

Recent CVE coverage

Dedicated modules and atomic tests target four February 2026 CVEs (RDP EoP, DWM privesc, MSHTML and SmartScreen bypasses) with patch checks and IoC enumeration.

Session abstraction (local + WinRM)

LocalSession runs via subprocess while WinRMSession uses pypsrp for remote PowerShell, with a common interface for cmd, PowerShell, registry, and file access.

Scan profiles and technique filters

Built-in quick, full, and stealth profiles plus per-tactic and per-technique filters let operators tune footprint and scope from the CLI or YAML config.

OS-aware execution

Modules declare SUPPORTED_OS and REQUIRES_ADMIN, and the engine auto-skips targets an assessment doesn't apply to across Win10/11 and Server 2019/2022.

03Architecture

A Click CLI (main.py) drives a central ScanEngine that discovers Python technique modules from the modules/ package tree and YAML atomic tests from atomics/, applies tactic/technique/profile filters, opens a Local or WinRM session, detects the target OS, then runs passive check() first and (only with --simulate) active simulate()+cleanup() plus uncovered YAML atomics. Results flow through data models into the reporter, MITRE mapper, and compliance mapper for HTML/JSON/CSV, ATT&CK Navigator, and CIS/NIST outputs, with an EvidenceLogger recording every action.

1
CLI (main.py, Click)
Entry point exposing scan, run-atomic, list-modules, and report commands with profile, tactic, technique, simulate, dry-run, and format options.
2
ScanEngine (core/engine.py)
Orchestrator that auto-discovers Python modules and YAML atomics, applies filters, sequences check/simulate phases, and produces a plan() for dry-run.
3
Python modules (modules/, BaseModule)
50 technique modules across 12 tactics, each implementing check(), simulate(), cleanup(), and get_mitigations() with declared SUPPORTED_OS and severity.
4
Atomic framework (core/atomic_runner.py + atomic_models.py)
Loads 226 YAML tests across 66 techniques, resolves input arguments and dependencies, executes via executor type, and runs cleanup commands.
5
Session layer (core/session.py)
BaseSession abstraction with LocalSession (subprocess) and WinRMSession (pypsrp) providing cmd, PowerShell, registry, and file operations.
6
Reporting & mapping (reporter, mitre_mapper, compliance_mapper)
Generate HTML/JSON/CSV reports, ATT&CK Navigator v4.5 layers, and CIS Benchmark v8 / NIST 800-53 compliance mappings.
7
Config & evidence (core/config.py, core/logger.py)
YAML settings, profiles, and technique toggles feed ScanConfig, while structlog and EvidenceLogger persist a timestamped audit chain.

05Security Controls

Credential Access (TA0006)
Modules for LSASS memory (T1003.001), SAM (T1003.002), NTDS.dit (T1003.003), LSA Secrets (T1003.004), Kerberoasting (T1558.003), credentials-in-files (T1552.001), and brute force / password policy (T1110).
Privilege Escalation (TA0004)
UAC bypass (T1548.002), access token manipulation (T1134), DLL search-order hijack and side-loading (T1574.001/.002), plus RDP EoP (T1210) and DWM SYSTEM privesc (T1068) CVE modules.
Defense Evasion (TA0005)
Disable security tools (T1562.001), disable event logging (T1562.002), modify registry (T1112), masquerading (T1036), clear event logs (T1070.001), and MSHTML/SmartScreen bypass CVE modules (T1218 / T1553.005).
Discovery (TA0007)
Eight modules plus the largest atomic set covering system info, account/permission-group, network service/config/connections, process, and file/directory discovery.
Persistence (TA0003)
Scheduled tasks (T1053.005), registry Run keys (T1547.001), file-association hijack (T1546.001), and Windows service creation with unquoted-path / writable-binary audit (T1543.003).
Lateral Movement & Impact
RDP/SMB/WinRM config and Pass-the-Hash audits (T1021.*, T1550.002); Impact modules for ransomware-readiness including inhibit recovery (T1490), service stop (T1489), shutdown (T1529), and encryption (T1486).
February 2026 CVE coverage
Four dedicated CVE modules with patch checks, IoC detection, and config enumeration: CVE-2026-21533 (RDP EoP), CVE-2026-21519 (DWM), CVE-2026-21513 (MSHTML), CVE-2026-21510 (SmartScreen).
Safety controls
Check-only default, explicit --simulate gate, --dry-run plan with zero target contact, SAFE_MODE modules that never simulate, OS auto-skip, mandatory cleanup, and a timestamped evidence audit chain.
Mitigation guidance
Every Python module implements get_mitigations() returning MITRE M-code remediations that surface as collapsible detail sections in the HTML report.

07Quick Start

$ git clone https://github.com/Krishcalin/Windows-Red-Teaming.git && cd Windows-Red-Teaming && pip install -r requirements.txt
$ python main.py scan --target 192.168.1.10 --profile full --simulate --dry-run   # preview the plan, no contact
$ python main.py scan --target localhost --profile quick   # passive, production-safe check
$ python main.py scan --target 192.168.1.10 --profile full --simulate   # active simulation with auto-cleanup
$ python main.py run-atomic --target localhost --technique T1059.001 --format json   # run atomics for one technique
$ python main.py scan --target localhost --format html --output report   # generate HTML report + ATT&CK layer
